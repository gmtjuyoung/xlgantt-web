# XLGantt 멀티프로젝트 + 인증 + 권한 설계안

## 1. 개요

| 항목 | 현재 | 목표 |
|------|------|------|
| 데이터 | Zustand 인메모리 | Supabase PostgreSQL |
| 인증 | 없음 | Supabase Auth (이메일/비번) |
| 프로젝트 | 단일 | 멀티 (생성/삭제/이동) |
| 권한 | 없음 | owner/admin/editor/viewer |

---

## 2. DB 스키마 (핵심 테이블)

```
profiles ─────┐
  id (= auth.users.id)  │
  email, display_name   │
                         │
projects ────────────────┤
  id, name, owner_id    │
  start/end_date        │
                         │
project_members ─────────┘
  project_id, user_id
  role (owner|admin|editor|viewer)

tasks ── dependencies
  project_id 기준

companies ── team_members ── task_assignments
  project_id 기준
```

### 권한 매트릭스

| 기능 | owner | admin | editor | viewer |
|------|:-----:|:-----:|:------:|:------:|
| 프로젝트 삭제 | O | | | |
| 설정/멤버 관리 | O | O | | |
| 작업 편집 | O | O | O | |
| 조회 | O | O | O | O |

---

## 3. 라우팅

```
/login              로그인
/signup             회원가입
/projects           프로젝트 대시보드 (카드 목록)
/projects/new       프로젝트 생성
/projects/:id       프로젝트 작업공간 (간트/진척/분석 등)
/projects/:id/settings  설정 + 멤버 관리
```

## 5. 구현 단계

| Phase | 내용 | 기간 |
|-------|------|------|
| **1** | 인증 (Auth + Login/Signup + Guard) | 1주 |
| **2** | DB 스키마 + RLS (Migration SQL) | 1주 |
| **3** | 서비스 레이어 + Store 전환 (인메모리→Supabase) | 1.5주 |
| **4** | 프로젝트 대시보드 + 전환 | 1주 |
| **5** | 권한 관리 (역할별 UI 제한) | 1주 |

---

## 6. Store 전환 전략

- 기존 store 시그니처 유지, 내부만 Supabase API로 교체
- 낙관적 업데이트: 로컬 먼저 변경 → 비동기 Supabase 저장 → 실패 시 롤백
- 프로젝트 전환: `switchProject(id)` → 해당 프로젝트의 tasks/deps/resources 로드
